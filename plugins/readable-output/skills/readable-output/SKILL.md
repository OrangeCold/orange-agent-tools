# readable-output

在你（agent）**分析和输出**方案、设计文档、技术方案、需求文档、计划、会议纪要、周报、报告、教程、规范等结构化长内容时，把下面的易读性原则当作**思考和组织的默认习惯**——不是「先写完一版再回头改」，而是从你构思核心结论、分组要点的那一刻起就照着来，一次成型。目标：让读者能快速 **Find（找到）→ Understand（理解）→ Use（使用）**。

方法论来自权威来源并经过对抗式验证。每条规则的出处、可信度与边界见 `${CLAUDE_PLUGIN_ROOT}/references/methodology.md`（需要向他人解释「为什么这么改」、或判断某条规则是否适用边界场景时再读，日常输出用本文件即可）。

## 为什么是「思考时带着」而不是「写完再改」

这个 skill 在你**开始任务时**就把原则加载进了上下文，所以它应当影响你整个生成过程——从想清楚核心结论的那一刻起，而不是等输出完一版再套。

「先写一版随意的、再按 skill 改一版好读的」是反模式，原因有三：

1. **双倍工作**：同一份内容生成两次。
2. **上下文污染**：两版内容同时留在上下文里互相干扰，反而降低最终质量。
3. **治标不治本**：大多数易读性问题（结论不清、结构混乱、术语漂移）根源在**分析阶段没想清楚**，事后润色弥补不了，一开始就想对才彻底。

所以正确姿势是：**分析和构思时就用这些原则自问**，让易读性成为思考的一部分；写，只是把已经想清楚的结构落成文字。

## 边界：做什么、不做什么

**做**：只优化内容**本身的易读性**——结构、句法、用词、术语。
**不做**：
- 不改变输出的**格式与载体**（不转 HTML、不转飞书、不加 Mermaid 图表、不动代码块）。你本来要输出 markdown 就输出 markdown，要写文件就写文件。
- 不改变**语义**——不增删信息、不改变方案的结论与立场、不补造原文/原方案没有的内容（这是红线，见下）。

这条边界让本 skill 与手动触发的 `easy-read`（把杂乱内容重做成 HTML/飞书）区分开：你是「思考与输出时的易读性习惯」，不是「重做载体」。

## 何时触发、何时跳过

自动触发是本 skill 的价值，但也容易过度。先判断这一条输出是否属于「需要易读性的长内容」：

**触发**（通常 ≥约 300 字，或含多个要点 / 多个章节 / 多个步骤的结构化内容）：
方案、设计文档、技术方案 / 技术设计、PRD / 需求文档、实施计划、会议纪要、周报 / 复盘、调研 / 分析报告、操作教程、规范指南、架构说明、对比选型……

**跳过**（套了反而啰嗦、降低信噪比）：
一两句话的问答、纯事实陈述、执行结果（"已创建 X.md"）、代码与命令输出、单步操作指引、闲聊。

> 判断不准时，问自己：读者需要「找要点、理解结构、照着用」吗？需要→带着意识写；只是回个事实→跳过。

## 核心方法：分析与组织时就带着这 6 点

这些**不是**僵化的线性流水线，也**不是**写完再套的检查清单——而是你在**分析和组织信息时带着的思维习惯**。构思方案时用它们自问，写出来自然就好读：

- 核心结论是什么？它**前置**了吗？
- 要点**超过 4 个**了吗？该**分层**了吗？
- 有没有**超长句**、多重嵌套？能拆短吗？
- 用**主动语态**了吗？**删冗词**了吗？
- 术语**统一**了吗？首次**释义**了吗？
- 标题**说清楚**内容了吗？

下面是每一点的展开与 why：

### 1. 结论前置（倒金字塔）

把**最重要、读者最想先知道**的信息放在最前面，细节按重要性递减往后排。并且这个原则贯彻到**每个层级**：文档第一段给结论 / 核心判断；每个小节开头一句话点明本节要点；每个列表项把关键信息放句首。

**为什么**：读者扫读时只看开头。结论埋在第三段，大多数人读不到。倒金字塔让读者用最低成本建立心理模型，也允许从后往前裁剪而不丢要点——这正是方案/文档被快速浏览时的真实场景。

### 2. 信息分块（chunking）

长内容拆成小块，一组**不超过 4 个**要点（认知科学：工作记忆约 4 个槽位，Cowan 修正了 Miller 早期的 7±2）。要点超过 4 个时，先**归类分层**（抽象出上级类别），而不是平铺一个 8 项的列表。

**为什么**：人按「组」而非「条」记忆。把 8 个散点归成 3 类各 2–3 点，读者吸收成本骤降。分块不只是视觉分隔，它表达的是信息之间的关系和层级。

### 3. 一段一旨，主谓相邻

一个段落只讲**一个主题**；一个句子尽量只表达**一个完整意思**。长句（中文单句超过约 60 字、或含多重「的/地」嵌套从句）拆成并列短句。让主语和谓语靠近，别隔着一长串修饰。

**为什么**：读者读到句尾时，句首的主语早已被挤出工作记忆，理解断裂。短句 + 主谓相邻 = 最低的理解摩擦。

### 4. 主动语态，删冗词

优先主动语态（「系统调用 X」而非「X 被系统调用」）；删掉不承载信息的冗词（「进行一个…的操作」「基于…的角度来看」「值得注意的是」「在…的情况下」）。日常用词优先于生僻词。

**为什么**：被动语态隐藏动作主体、拉长句子、增加理解负担；冗词稀释信息密度。主动 + 精炼让每句话都「有内容」。

### 5. 术语一致，首次释义

同一个概念全文用**同一个叫法**（单一事实来源）；术语**首次出现时**用一句话解释，或附上对照。避免同一事物一会儿叫「订单中心」一会儿叫「订单服务」。

**为什么**：术语漂移会让读者怀疑「这是不是两个东西？」，反复对照消耗注意力。首次释义 + 全文统一 = 读者一次建立映射，后续畅通。

### 6. 描述性标题与层级

用**说清楚内容**的标题（「三、降级策略：主库故障时切只读」），而非模糊标题（「三、其他」）。建立清晰的标题层级（H1/H2/H3），让目录本身就是一份摘要。

**为什么**：标题是读者导航的锚点，也是扫读时唯一会被逐字看的内容。描述性标题让读者不点进去就知道这节讲什么。

## 红线：绝不逾越

这些是调研报告对抗式验证后明确的禁区，违反会损害内容的可信度：

1. **不增删语义**：易读性优化只改「怎么说」，不改「说什么」。方案原本有 3 个风险，优化后还是 3 个；不能为了「精炼」砍成 2 个，也不能为了「完整」补成 4 个。
2. **不补造**：原文/原方案没有的事实、数据、结论，绝不替用户编造。拿不准的，要么问用户，要么标注为「待确认」，绝不发挥想象。
3. **不把可读性公式当判据**：Flesch-Kincaid / Gunning Fog 等公式依赖音节计数，对中文无效，且只测表层特征。不要因为「公式分数不够」就机械改写。结构特征（长句、堆叠、术语漂移）才是真正的难读信号。
4. **不把维度当线性流水线**：不要僵化地「先做 A 再做 B 再做 C」。这些维度是并行体检的检查项，按实际内容哪里有问题就改哪里。

## 兜底自检（写完后快速扫一遍）

哪怕思考时已经带着意识，也难免有遗漏。写完后用下面这条清单**快速扫一眼**，补掉构思时漏掉的结构问题——它是**安全网，不是主流程**，不必逐项重写，只在发现明显缺口时回头补：

- [ ] **第一段/第一句**是否已给出结论或最重要信息？（倒金字塔）
- [ ] 是否有**清晰的标题层级**？标题是否**描述性**？（导航锚点）
- [ ] 长段落是否拆成了 **≤4 个**信息块？长列表是否分层了？（chunking）
- [ ] 是否**一段一旨**、主谓相邻、主动语态、无超长句？（句法）
- [ ] 术语是否**全文一致**、首次出现已**释义**？（术语）
- [ ] 读者能否 **Find → Understand → Use**？（Plain Writing Act 三维度）
- [ ] **忠实度**：有没有在组织过程中增删语义、补造内容？（红线兜底）

## 工作流

1. **判断**：这次输出是「需要易读性的结构化长内容」吗？否→不套本 skill，正常输出。
2. **带着意识分析与构思**（关键）：在理清方案内容的同时，就用上面 6 点自问——核心结论是什么（它要前置）？要点超过 4 个了吗（要先分层）？有没有超长句、术语漂移？让易读性成为思考的一部分，而不是写完之后的额外动作。
3. **一次成型**：直接按上面的组织方式写出来。**不要**先写一版「随意草稿」再改成「易读版」——那是双倍工作，还会让上下文塞进两个版本互相污染。
4. **兜底自检**：写完后用清单快速扫一眼，补掉漏掉的结构问题（安全网，不是主流程）。
5. **输出**：保持原本的格式与载体，把内容作为最终输出。
