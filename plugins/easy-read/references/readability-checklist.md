# 易读性自检清单（写入产物前过一遍）

> easy-read 两个 skill（html / feishu）共享的验收清单。在生成最终产物（HTML / 飞书文档）**之前**过一遍，确认整理后确实更易读。源自 Plain Writing Act 的 Find / Understand / Use 三维度 + chunking + 一致性。

## 三维度验收（Find → Understand → Use）

### Find（读者能找到）
- [ ] 第一屏 / 顶部是否已给出**核心结论**？（倒金字塔：结论先行）
- [ ] 标题是否**描述性强**（看标题就知道讲什么），而非“概述”“其他”这类空标题？
- [ ] 层级是否清晰（H1 → H2 → H3），读者能靠标题快速定位？

### Understand（读者能理解）
- [ ] 每个要点能否**独立理解**，不需要回头翻上文？
- [ ] 专业术语首次出现是否已**释义**？缩写是否展开？
- [ ] 是否还有**超长句 / 超长段落**漏网？（单句 >60 字、单段堆 >5 要点需拆）
- [ ] 机制 / 决策 / 流程类**长段落（>3 行）是否前置了 2–4 个要点 TL;DR**？（见 content-structuring「段落要点化」）
- [ ] 关系 / 顺序 / 结构是否已**图表化**（比文字更快讲清的才画）？

### Use（读者能用）
- [ ] 读者能否据此**行动 / 决策**？下一步 / 待办是否清晰？
- [ ] 需要读者做的事项是否落成**待办 / 清单**，而非埋在正文里？

## 结构与一致性

- [ ] 每个 H2 下的**并列要点 ≤4 个**？超过是否已归并或下沉一层？（chunking）
- [ ] **术语全文统一**（同一概念只用一个词）？术语密集处是否有对照表？
- [ ] 是否**满篇彩色 / 满篇 emoji**？（应克制——强调才用）

## 忠实性兜底

- [ ] 有没有**替用户下结论、补造数据 / 事例**？（应无）
- [ ] 有没有**改变原意、把“可能”说成“一定”**？（应无）
- [ ] 拿不准的地方是否走了澄清步骤问用户，而非自行发挥？

> 任一项不达标 → 回结构化 / 澄清步骤修正，再产出。不要带着已知缺陷输出。
